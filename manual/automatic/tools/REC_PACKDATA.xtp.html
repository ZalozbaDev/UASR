Description

REC_PACKDATA.xtp compiles binary data files for the dLabPro recognizer. There are two modes:

  1. REC_PACKDATA.xtp dlg packs data files for a dialog. This includes a finite dialog state machine and exactly one finite state recognition network (HMMs* o lexicon* o language model) for each dialog state.
  2. REC_PACKDATA.xtp rec packs data files for a single grammar (i. e. a dialog with only one state). In this case there is only one finite state recognition network.

Details on input and output files can be found in the documentation of the commands.

See also

  • Writing grammar files
  • Writing dialog files

Configuration

HMM.xtp is configured through a UASR configuration file (parameter $2 ). The configuration file is a Java properties file containing key-value-pairs. The common configuration keys are described with the utility script cfg.itp (see there). Some commands use dedicated configuration keys which are described with the command. Dedicated keys may be placed in the configuration file just like all other keys. Any key (common and dedicated) may be specified or overwritten through the command line using -P<key>=<value> options: 

 dlabpro REC_PACKDATA.xtp dlg mydialog.cfg -Pskip=FG -Prender=yes

-P options always overwrite the configuration file. The heading " uasr. " of the keys may be omitted in -P options.

Writing Dialog Files

A dialog file is an ASCII file defining a finite state dialog model, a lexicon and a exactly one finite state language model for each dialog state. Dialog files are organized by lines. The line format is:

 <PREFIX>: information

Blank lines and lines starting with the hash character (#) are ignored. White spaces are significant. Sequences of white spaces – except line breaks – behave like a single space. The following prefixes are supported:

Prefix Description
DLG: One dialog rule
LEX: One lexicon entry
GRM.<dlg-state>: One grammar rule of the dialog state named <dlg-state>
GRM: One grammar rule shared by all dialog states

The syntax of lexicon entries and grammar rules is explained in section "Writing grammar files". The format of dialog rules (prefix DLG:) is identical to the format of grammar rules. However

  • every dialog state must be named, and
  • terminal transitions must not contain output symbols, weights or stack tokens.

The input symbol of a terminal dialog transition should be enclosed in double underscores, e. g. __SLEEP__. If the recognition result is equal to the terminal of a dialog transition starting at the current dialog state, the dialog state is changed by traversing this transition. If no suitable dialog transition is found, the recognizer remains in the current dialog state. If more than one suitable dialog transition is found, a random state change occurs (should be avoided!).

The start state (S) of the dialog is the default or "sleeping" state. The dLabPro recognizer can be configured to fall back to this state after a certain time of inactivity.

Example

This ready-to-go example defines a dialog with a sleeping state and an active state which conatains a micro-local grammar for sepcifying the dimensions of a rectangle in meters:

 dialogexample.grm.txt 
## Dialog rules
DLG: (S)       __WAKEUP__ (1)
DLG: (1)       __SLEEP__ (S)

## Grammar rules of dialog state (S)
GRM.S: (S)     <PAU> (S)
GRM.S: (F)     <PAU> (F)
GRM.S: S       computer:__WAKEUP__

## Grammar rules of dialog state (1)
GRM.1: (S)     <PAU> (S)
GRM.1: (F)     <PAU> (F)
GRM.1: S       ok:__SLEEP__
GRM.1: S       SIDE LENGTH
GRM.1: SIDE    breite:breite
GRM.1: SIDE    laenge:laenge
GRM.1: LENGTH  DIG (A) komma:komma (B) DIG (C) meter:meter
GRM.1: (A)     (C)
GRM.1: (C)     (B)
GRM.1: DIG     null:null
GRM.1: DIG     eins:eins
GRM.1: DIG     zwei:zwei
GRM.1: DIG     drei:drei
GRM.1: DIG     vier:vier
GRM.1: DIG     fuenf:fuenf
GRM.1: DIG     sechs:sechs
GRM.1: DIG     sieben:sieben
GRM.1: DIG     acht:acht
GRM.1: DIG     neun:neun

## Lexicon entries
LEX: acht      (Q|)axt
LEX: breite    braIt@
LEX: computer  (k|g|)(O|u:|o:|)(m|n|N|)(p|b|)(C|S)u:(t|d)(a:|)(6|)(x|)
LEX: eins      (Q|)aIns
LEX: drei      draI
LEX: fuenf     fY(n|m)f
LEX: komma     kOma:
LEX: laenge    lEN@
LEX: meter     me:t6
LEX: neun      nOYn
LEX: null      n(U|u:)l
LEX: ok        o:ke:
LEX: sechs     zEks
LEX: sieben    zi:(b(@|)n|m)
LEX: vier      fi:6
LEX: zwei      tsv(aI|o:)

Note that – in contrast to the similar example in section "Writing grammar files" – every terminal transition definition includes an input and an output symbol (both lexicon entries). This is required because the transition ok:__SLEEP__ (rule three in GRM.1) defines an output symbol which is no lexicon entry but triggers a dialog state change. Now that there is one non-empty output symbol, REC_PACKDATA.xtp will not automatically fill in other output symbols. Hence, the other output symbols must be included in the grammar rules.

uasr.am.model  = "3_15"

## REC_PACKDATA keys
uasr.out       = "./output"
uasr.dlg.fsg   = "dialogexample.grm.txt"

and

Writing Grammar Files

A grammar file is an ASCII file defining a lexicon and a finite state language model. Grammar files are organized by lines. The line format is:

 <PREFIX>: information

Blank lines and lines starting with the hash character (#) are ignored. White spaces are significant. Sequences of white spaces – except line breaks – behave like a single space. The following prefixes are supported:

Prefix Description
LEX: One lexicon entry
GRM: One grammar rule

Lexion Entries

Each lexicon entry maps one orthograpic to one phonetic string (i. e. a string of HMM names). The format of a lexicon entry is: 

 LEX: <orthograpic> <phonetic>

Orthographic strings are arbitrary. They must, however, not contain white spaces, colons (:) or commas (,). It is recommendable to avoid special characters – except underscores (_) – altogether.

Phonetic strings are sequences of HMM names. They must not conatain white spaces. Phonetic strings may contain variants in the form

 <prefix>(<variant1>|<variant2>[|...])<suffix>

A variant can be empty or contain nested veriants. For more information see documentation of dLabPro method fst -compile. For instance, the lexicon entry 

 LEX: sieben zi:(b(@|)n|m)

will expand to the pronunciation variants zi:b@n, zi:bn, and zi:m. Orthograpic strings may occur multiply. Hence the tree entries

 LEX: sieben zi:b@n
 LEX: sieben zi:bn
 LEX: sieben zi:m

are equvialent to the previous example. See UASR Phoneme Sets for phoneme tables.

Grammar Rules

A grammar rule has the format:

 GRM: <left-symbol> <right-symbol>[ <right-symbol> [...]]

where <left-symbol> is a named state or a non-terminal transition and <right-symbol> is a named state, a terminal, or a non-terminal transition. Symbol strings must not contain white spaces, colons (:) or commas (,). Though it is recommendable to avoid any special characters, parenthesis (()), brackets ([]) and periods (.) are explicitly allowed.

  1. Named states:
    An arbitrary string enclosed in parentheses, e. g. (A). The names (S) and (F) are reserved and denote the start and final states.
  2. Non-terminal transitions:
    An arbitrary string not enclosed in parentheses, e. g. A. The non-terminal S is reserved and denotes the start rule.
  3. Terminal transitions:
    A string in the format
      
     <in>[:<out>[:<weight>[:<token>]]]

      with
     
    Element Description
    <in> terminal input symbol; must be an orthographic string defined in the lexicon
    <out> – optional – terminal output symbol; the recognition result is a string of these symbols
    <weight> – optional – a logarithmic transition weigt; a transition with an infinite weight (e. g. ::INF) is not added to the grammar. However, the path up to the transtion's start state und from the transition's end state will be correctly inserted. Rules with infinite weight are useful in connection with stack tokens in order to re-use previosly addad paths in the grammar.
    <token> – optional – a stack token in the format +<id> or -<id>, where the first pushes the token named <id> to the decoder stack and the second pops the same token from the stack.

    Example:
     
     sieben:val=7::+SEVEN

Depending on the <left-symbol> there are two types of grammar rules:

  1. Regular rules:
    The left symbol is a named state. Such rules are expanded by appending a path of (named) states and transitions to the named state on the rule's left side. If the last element does not denote a named state, the last generated transition will terminate at the final state (F). Example:
     
     GRM: (S) RECTDIM (A) LENDIM (F)
  2. Context free rules:
    The left symbol is a non-terminal. All non-terminal transitions named after the left symbol side will be replaced by the path defined by the sequence of right symbols. Example:
     
     GRM: S RECTDIM LENDIM

In both cases the rule expansion will properly regard named states in the rule. That is, if the state exists it will be used and if it does not exist, it will be created.

Example

This ready-to-go example defines a micro-local grammar for sepcifying the dimensions of a rectangle in meters:

 grammarexample.grm.txt 
## Grammar rules
GRM: (S)     <PAU> (S)
GRM: (F)     <PAU> (F)
GRM: S       SIDE LENGTH
GRM: SIDE    breite
GRM: SIDE    laenge
GRM: LENGTH  DIG (A) komma (B) DIG (C) meter
GRM: (A)     (C)
GRM: (C)     (B)
GRM: DIG     null
GRM: DIG     eins
GRM: DIG     zwei
GRM: DIG     drei
GRM: DIG     vier
GRM: DIG     fuenf
GRM: DIG     sechs
GRM: DIG     sieben
GRM: DIG     acht
GRM: DIG     neun

## Lexicon entries
LEX: acht    (Q|)axt
LEX: breite  braIt@
LEX: eins    (Q|)aIns
LEX: drei    draI
LEX: fuenf   fY(n|m)f
LEX: komma   kOma:
LEX: laenge  lEN@
LEX: meter   me:t6
LEX: neun    nOYn
LEX: null    n(U|u:)l
LEX: sechs   zEks
LEX: sieben  zi:(b(@|)n|m)
LEX: vier    fi:6
LEX: zwei    tsv(aI|o:)

The first three grammar rules are start rules. Rules one and two include a pause (pre-defined lexicon entry <PAU>) of arbitrary length before and after the utterance. It is recommended to always include these two rules. Rule three is the actual start rule.

To compile this grammar you need model files (see section "Input and Output Files" above) and a UASR configuration file. We assume these files are organized as follows:

 <current folder>

 3_10_mod.vad  VAD model file (not required by REC_PACKDATA.xtp but used by the dLabPro recognizer)
 3_15.hmm  HMM file
 feainfo.object  Feature information file
 output  Output files (folder will be created, existing files will be overwritten without warning)
 3_15.gmm   Gaussian mixture model file
 feainfo.object   Copy of feature information file
 GRM.svg   Rendering of grammar transducer (only if uasr.render=yes)
 sesinfo.object   Session information file
 grammarexample.cfg  Configuration file for REC_PACKDATA.xtp rec (see below)
 grammarexample.grm.txt  Example grammar as ASCII file
 recognizer.cfg  Configuration file for the dLabPro recognizer (see below)

Then the UASR configuration file for REC_PACKDATA.xtp should look like this:

 grammarexample.cfg 
## UASR configuration
uasr.dir.model = "./models"
uasr.am.model  = "3_15"
 
## REC_PACKDATA configuration
uasr.out       = "./output"
uasr.lm.fsg    = "grammarexample.grm.txt"

To compile the grammar change to the root folder of the directory structure outlined above and invoke:

REC_PACKDATA.xtp rec grammarexample.cfg -Prender=yes

The last argument is optional and will render der grammar transducer as SVG graphics (requires the AT&T dot tool). Fig. 1 shows the rendering.

To test the grammer you need a configuration file for the dLabPro recognizer:

 recognizer.cfg 
## Recognizer configuration
data.feainfo = ./output/feainfo.object
data.gmm     = ./output/3_15.gmm
data.vadinfo = ./models/3_10_mod.vad

Invoke the dLabPro recognizer in live mode using the command line:

recognizer -cfg recognizer.cfg

Recognition results are concatenations of the output symbols of the decoded paths through the grammar. If the grammar does not define any output symbols (as in our example), the terminal input symbols will be used as output.

Semantic output

The terminal output symbols of the grammar definition may be used to generate feature value trees. For details see KT-Wiki/UASR (Project)/Serialisation of FVR's (KT-Wiki account required). The following version of the example grammar produces such semantic output:

 grammarexample-fvr.grm.txt 
## Grammar rules
GRM: (S)     <PAU> (S)
GRM: (F)     <PAU> (F)
GRM: S       :FVR[RECTDIM SIDE LENGTH :]
GRM: SIDE    :[side=width breite :]
GRM: SIDE    :[side=length laenge :]
GRM: LENGTH  :[val= DIG :] (A) :[val=. komma :] (B) :[val= DIG :] (C) :[unit= meter:m :]
GRM: (A)     (C)
GRM: (C)     (B)
GRM: DIG     null:0
GRM: DIG     eins:1
GRM: DIG     zwei:2
GRM: DIG     drei:3
GRM: DIG     vier:4
GRM: DIG     fuenf:5
GRM: DIG     sechs:6
GRM: DIG     sieben:7
GRM: DIG     acht:8
GRM: DIG     neun:2 
 
## Lexicon entries
LEX: acht    (Q|)axt
LEX: breite  braIt@
LEX: eins    (Q|)aIns
LEX: drei    draI
LEX: fuenf   fY(n|m)f
LEX: komma   kOma:
LEX: laenge  lEN@
LEX: meter   me:t6
LEX: neun    nOYn
LEX: null    n(U|u:)l
LEX: sechs   zEks
LEX: sieben  zi:(b(@|)n|m)

Function -PACK_sesinfo Functions Script Package Keyword Index Function Index Home
    Signature :    -PACK_sesinfo(retSI,retDlg)
    Source :    tools/REC_PACKDATA.xtp(586)

Description

Packs the session info file.

The session information object is structured as follows:

iSI Description
  idLMtos – deprecated – The output symbols of the recognition network, to be replaced by itRN.os.
  itGP The Gaussian-to-phoneme labelling transducer. Used by the recognizer to generate phonetic output strings. For general acoustic pattern recognition applications read "HMM labels" instead of "phonemes".
  itLX The lexicon (phoneme-to-word) transducer. The object contains exactly one FST unit per lexicon entry. For general acoustic pattern recognition tasks the "lexicon" consists of application specific re-occurring HMM label sequences.
  itRN The recognition network translating between Gassian sequences and grammar output symbol strings.
  itRNr The reference recognition network translating between Gaussian sequences and phoneme strings. The object is used to obtain a secondary, unconstrained decoding for measuring the recognition confidence. For general acoustic pattern recognition applications read "HMM labels" instead of "phonemes". The object is only needed if a reference decoding is performed, otherwise it may be omitted.
