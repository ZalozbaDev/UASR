Configuration

SVM.xtp is configured through a UASR configuration file (parameter $2). The configuration file is a Java properties file containing key-value-pairs. The common configuration keys are described with the utility script cfg.itp (see there). Some commands use dedicated configuration keys which are described with the command. Dedicated keys may be placed in the configuration file just like all other keys. Any key (common and dedicated) may be specified or overwritten through the command line using -P<key>=<value> options:

 dlabpro SVM.xtp trn $UASR_HOME/config/vm/VMV.cfg -P...

-P options always overwrite the configuration file. The heading "uasr." of the keys may be omitted in -P options.

For further information

  • see documentation of util/cfg.itp to learn more about UASR's organization of databases.
  • see documentation of -CFG_get_path to learn how UASR determines paths to the various data sets.
  • see documentation of -CFG_get_flist to learn how UASR obtains file lists of the various data sets.

Description

SVM.xtp trn is part of the UASR acoustic model training outlined in figure 1. It creates a set of SVMs from either

  1. a database consisting of labelled primary feature vector files or
  2. a database consisting of signal files and labels.

(See documentation of util/cfg.itp for details on the organization of databases in UASR.)

The first first method requires more disk space but is quicker if there is more than one HMM training iteration. The primary feature vector files database must be created before the actual model training

  • using FEA.xtp ana (primary feature analsis of signal files) or
  • using FEA.xtp imp (import of external feature files).

The second method saves the disk space for storing the primary feature vectors but it requires to compute these for each SVM training anew.

Remark on Incremental SVM Training

The current incremental training creates a random partitioning of the training set before training. This partitioning will be different each time SVM_training.xtp is executed. Hence the results of the incremental SVM training are not reproducible. Results of the conventional training are, however, reproducible. The incremental training is activated by specifying a maximal training partition size through the configuration key uasr.svm.incr.

This flaw is to be fixed!

SVM.xtp fpar Script Package Keyword Index Function Index Home
    Signature :    SVM.xtp fpar <cfgfile>
    Synopsis :    SVM findparam

Description

SVM.xtp fpar implements the method to find optimal parameters for a SVM given a specific dataset. The procedure is prosed from the authors of libsvm and explained in the guide to libsvm section 3.2: Guide.pdf.

The procedure is implemented for the RBF-Kernel. The parameters to find are C and gradient. Other parameters for other kernels could be find the same way, but are not yet considered here.

There es a loop for every parameter, thus a beginning, ending and stepwidth for every parameter is needed. For every parametercombination a crossvalidation on the hole dataset (all.flst) is calculated finding an accuracy. Best parameters are those with the highest accuracy. Logfile findparam.dn3 is written or continued.

Description

Compiles a data set for SVM training or classification.

Parameters

   data    idF   Turn list
   string    sSns   Sensor ID, NULL or empty for one-channel recordings (typical case for speech recognition)
   string    sSfa   Secondary feature analysis, empty if not specified
   bool    bSig   TRUE to use signal data as vectors, FALSE to use primary feature vectors
   bool    bSv   TRUE use supervector, FALSE no supervector, every feature vector is taken (makes sense when bSig=FALSE)
   data    idLtb   Label table
   data    idDts   The compiled data set where each item of the data set is handled as
  • bSig=TRUE:
    one vector containing the signal data or
  • bSig=FALSE:
    one supervector made by concatenating the feature vectors
  • bSv=FALSE:
    each feature vector ist taken
followed by a numeric label for each item (or feature vector) of the data set.

Return value

TRUE if successful, FALSE otherwise
