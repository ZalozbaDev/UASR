Synopsis

Utility functions: Labels

Label component format

In UASR the label convension is that every feature vector has a label. So no labels will be lost by working with these vectors. The disadvantage is that we lose the boundaries of one label after another identical one. So there is a configuration option "uasr.fea.labidx" which enables label indices. With this options the labels looks like "NAME[INDEX]".

To support these labels the function data.-gen_index and statistics.-update ignores the indices in brackets if the option /label is used.

Warning: UASR supports that label format only for FEA.xtp and HMM.xtp

Function -LAB_get_map Functions Script Package Keyword Index Function Index Home
    Signature :    -LAB_get_map
    Source :    util/lab.itp(142)

Description

Returns the database-2-UASR label map. The map is a data instance containing two columns: (0) the database dependent label and (1) the UASR label mapped to the database dependent label.

The function proceeds as follows:

  1. Try to load the ASCII file named "labmap.txt" located in the info directory or specified by the configuration key <uasr.lab.map>. If either of these files exists, return a map containing its contents.
  2. Call -LAB_get_tab and a data instance containing two identical components mapping the UASR labels to themselves.

Return value

The label map (data instance)

Access to Configuration and Global Instances

   <uasr.lab.map>    read

Function -LAB_import Functions Script Package Keyword Index Function Index Home
    Signature :    -LAB_import(sDir,sTid,idLab)
    Source :    util/lab.itp(219)

Description

Abstract method for signal label import. The function must be overwritten depending on the database, for example: 
 
  /disarm -LAB_import_dummy /disarm -LAB_import =;

The result instance idLab must be filled with an RLE-compressed label stream and have the following format:

ComponentDescription
No.Type
0stringLabel name
1longLabel position (samples)
2longLabel length (samples)

The result instance may also have only one string component and one record. In this case, the only label idLab[0,0] is used for the entire signal.

See also

-SIG_label

Parameters

   string    sDir   Path to label files
   string    sTid   Turn identifier. See -SIG_import for the format of this identifier
   data    idLab   Data instance to be filled with RLE-compressed label stream

Function -LAB_import_ascii Functions Script Package Keyword Index Function Index Home
    Signature :    -LAB_import_ascii(sDir,sTid,idLab)
    Source :    util/lab.itp(287)

Description

Label import from ASCII file. The file is expected to contain exactly one label entry per line. One entry has to consist of three white space separated fields: 1. the label name, 2. the label offset (in samples), and 3. the label length (in samples). Label files must have the extension ".lab".

This function may overwrite -LAB_import:

 
  /disarm -LAB_import_ascii /disarm -LAB_import =;

Structure of Result

ComponentDescription
No.Name
0<sLid>Label name
1POSLabel position (samples)
2CNTLabel length (samples)

Parameters

   string    sDir   Path to label files
   string    sTid   IZFP_RS measurement identifier. See -SIG_import for the format of this identifier
   data    idLab   Data instance to be filled with labels

Function -SNS_get_tab Functions Script Package Keyword Index Function Index Home
    Signature :    -SNS_get_tab
    Source :    util/lab.itp(658)

Description

Returns the database specific sensor ID table. The function proceeds as follows:
  1. Check the configuration for a key named uasr.sensor. If found, return a list containing one entry with the value of that key.
  2. Try to load the ASCII file named "sensors.txt" located in the info directory or specified by the configuration key <uasr.am.sensors>. If either of these files exists, return a list containing its contents.
  3. Return a list containing one empty entry indicating that there is only one sensor (which is typically the case for speech processing).

Return value

The sensor table (data instance)
